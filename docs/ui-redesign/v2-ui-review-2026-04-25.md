# V2 UI Review (2026-04-25)

Scope: `dashboard/v2.html`, `dashboard/v2-app.jsx`, and `dashboard/v2-data.js`.

## Overall

V2 is a strong upgrade over V1 visually and structurally. The card hierarchy is clear, performance tab plumbing is in place, and the adapter layer (`v2-data.js`) provides good backward compatibility with current pipeline output.

## High-priority findings

1. **Accessibility gap on date navigation pills (keyboard + semantics).**
   - `DateBar` renders clickable `<div>` elements without keyboard handlers or button semantics.
   - Mouse/touch users can change dates, but keyboard and assistive technology users lose this path.
   - Recommendation: switch each date pill to `<button type="button">` with an `aria-label`, and preserve current styling.

2. **Likely directionality bug in “Why this pick” support logic for Opponent K%.**
   - In `PickDetail`, `oppSupports` is computed as:
     - under pick => `oppDelta > 0`
     - over pick => `oppDelta < 0`
   - This appears inverted relative to baseball logic (higher opponent K% generally supports OVER K props).
   - Recommendation: reverse `oppSupports` condition so OVER aligns with positive opponent K delta and UNDER aligns with negative delta.

3. **Time-zone consistency risk between slate logic and rendered game times.**
   - `v2-data.js` uses a Phoenix-based day-roll strategy (`getAppDate`) while `v2-app.jsx` renders times with browser-local `toLocaleTimeString`.
   - Users outside Arizona can see game times in local timezone while date selection logic follows Phoenix date cutover.
   - Recommendation: standardize display timezone explicitly (e.g., force `America/Phoenix` or show timezone badge).

## Medium-priority findings

4. **Pass cards remain interactive but may not provide additional value.**
   - `PickCard` opens detail sheet for all verdicts, including `PASS`.
   - If that is intentional, keep as-is; if not, consider disabling detail open for PASS to reduce clicks.

5. **Desktop framing is polished, but fixed 414px may constrain larger-screen utility.**
   - Current desktop mode intentionally emulates mobile frame.
   - If power users review lots of cards on desktop, consider an optional wider layout toggle.

## What’s working well

- Strong visual hierarchy and readable stat scanline on cards.
- Nice resilience/fallback behavior in adapter (`normalizePitcher`) and steam derivation.
- Good UX touches: sticky header, themed tokens, result pills for past slates, and clear verdict badges.

## Suggested next implementation order

1. Accessibility fix on date pills.
2. Opponent K support-direction correction.
3. Timezone normalization decision (product call).
4. Optional desktop-width enhancement.
